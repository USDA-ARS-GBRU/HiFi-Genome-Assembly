Short answer: **yes, but indirectly** — **hifiasm does not print a single line saying “genome coverage = X×”**, but it *does* report all the information you need to calculate and sanity-check coverage, and it reports it in **multiple places in the logs**.

Here’s how to read it properly.

---

## 1️⃣ What hifiasm *does* report in the logs

### A. Read statistics (early in the log)

Early in the run you’ll see lines like (example):

```
[M::ha_analyze_count] collected 32,845,112,394 bases in 3,215,678 reads
[M::ha_analyze_count] median read length: 14,832
```

This tells you:

* **Total bases in HiFi reads**
* **Total number of reads**
* **Read length distribution**

This is your **numerator** for coverage.

---

### B. Estimated genome size (k-mer–based)

You will usually see something like:

```
[M::ha_hist] peak at k-mer multiplicity 28
[M::ha_hist] estimated genome size: 620,345,812
```

or similar phrasing depending on version.

This is hifiasm’s **k-mer–based genome size estimate**.

This is your **denominator** for coverage.

---

### C. Implicit coverage from k-mer peak

The **k-mer multiplicity peak** is effectively the *modal coverage* of homozygous sequence.

For example:

```
k-mer peak at ~28
```

means:

* effective HiFi depth ≈ **28×**

This is often the **most reliable coverage indicator** for HiFi assemblies.

---

## 2️⃣ How to compute coverage explicitly

You compute coverage yourself as:

[
\text{Coverage} = \frac{\text{total HiFi bases}}{\text{estimated genome size}}
]

Example:

```text
Total HiFi bases = 32.8 Gb
Genome size      = 620 Mb

Coverage ≈ 52.9×
```

If this differs substantially from the k-mer peak:

* suspect high duplication
* suspect contaminants
* suspect collapsed repeats

---

## 3️⃣ Where coverage shows up *implicitly* later in the log

### A. Error correction / graph construction messages

You’ll see phrases like:

```
[M::ha_pt_gen] peak coverage: 31
[M::ha_pt_gen] cutoff coverage: 6
```

These are **coverage-derived thresholds** used internally by hifiasm.

They confirm:

* what hifiasm thinks is “normal depth”
* what it treats as low-coverage noise

---

### B. If heterozygosity is present (less likely for RILs)

You may see:

```
[M::ha_hist] heterozygous peak at ~14
[M::ha_hist] homozygous peak at ~28
```

That directly tells you:

* diploid structure
* effective coverage per haplotype

In RILs, you usually **won’t** see a strong heterozygous peak — which is expected.

---

## 4️⃣ What hifiasm does *not* do (important)

❌ It does **not** print:

```
Genome coverage: 47×
```

❌ It does **not** explicitly summarize:

* post-filter coverage
* coverage used after read trimming

So you must:

* extract values
* compute it yourself
* sanity-check against k-mer peaks

This is normal and expected.

---

## 5️⃣ Recommended “coverage extraction” checklist (for your notes)

When documenting assemblies, I recommend recording **three numbers** per RIL:

1. **Total HiFi bases** (from log)
2. **Estimated genome size** (from log)
3. **K-mer peak coverage** (from log)

Example:

```
RIL_01:
  HiFi bases:        31.2 Gb
  Genome size est.:  615 Mb
  Raw coverage:      ~50.7×
  K-mer peak:        ~27×
```

If those agree within ~20–30%, you’re in great shape.

---

## 6️⃣ One-liner to extract coverage info from logs

If your log is `hifiasm.log`:

```bash
grep -E "collected|genome size|k-mer|peak" hifiasm.log
```

That usually pulls all the relevant lines.

---

## 7️⃣ For RILs specifically (important context)

Because these are **highly inbred**:

* k-mer peak = *true* effective coverage
* no need to halve coverage like in heterozygous diploids
* coverage interpretation is cleaner than wild accessions

So if hifiasm reports:

```
k-mer peak at ~30
```

You can safely say:

> “~30× HiFi coverage”

---

## TL;DR

* ✅ **Yes**, hifiasm logs contain coverage information
* ❌ **No**, it does not print a single explicit “coverage” line
* 📌 Coverage is inferred from:

  * total read bases
  * k-mer genome size estimate
  * k-mer multiplicity peak
* 🧠 For RILs, the k-mer peak is the best coverage proxy
